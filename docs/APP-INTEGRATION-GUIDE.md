# Audiobook Integration Guide for App Developers

This document provides everything needed to implement audiobook playback with sentence-level text synchronization in the Noble Imprint mobile/desktop app. The audio generation pipeline already exists — your job is to consume the generated files and build the playback experience.

## What Already Exists

Audio is generated automatically by a pipeline that runs when book content changes. The pipeline produces three files per chapter, stored in Google Cloud Storage:

| File | Example | Purpose |
|------|---------|---------|
| **Chapter MP3** | `02-chapterone.mp3` | The narrated audio for that chapter |
| **Timestamps JSON** | `02-chapterone.timestamps.json` | Sentence-level start/end times mapped to text |
| **TTS JSON** | `02-chapterone.tts.json` | The preprocessed text that was sent to the TTS engine (for debugging) |

A **manifest** (`manifest.json`) indexes all chapters with their file names, durations, character counts, and content hashes.

All files are in a GCS bucket (`noble-imprint-audiobooks`) and are **not publicly accessible**. They must be accessed via signed URLs (1-hour expiry) generated by the website's API, or directly via GCS SDK with appropriate service account credentials.

## How to Determine if a Book Has Audio

Each book has a `meta.json` in the Resources repo. If the book has audiobook generation enabled, it contains:

```json
{
  "title": "Oration II",
  "subtitle": "In Defense of His Flight to Pontus",
  "audiobook": {
    "enabled": true,
    "skip_sessions": ["01-FrontMatter.md", "08-Bibliography.md"]
  }
}
```

**Check:** `meta.audiobook && meta.audiobook.enabled === true`

Sessions listed in `skip_sessions` do NOT have audio — don't show a play button for those.

## Fetching Audio Files

### Option A: Use the website API (simplest)

The website exposes two endpoints that handle GCS authentication and return signed URLs:

**Get the manifest:**
```
GET https://resources.noblecollective.org/api/audio/manifest/{bookPath}
```

Example:
```
GET https://resources.noblecollective.org/api/audio/manifest/A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II
```

Returns the full manifest JSON (see format below).

**Get a signed URL for any file:**
```
GET https://resources.noblecollective.org/api/audio/url/{bookPath}/{filename}
```

Example:
```
GET https://resources.noblecollective.org/api/audio/url/A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II/02-chapterone.mp3
```

Returns:
```json
{
  "url": "https://storage.googleapis.com/noble-imprint-audiobooks/audio/a-library-of-classics/a-pastoral-shelf/oration-ii/02-chapterone.mp3?GoogleAccessId=...&Expires=...&Signature=..."
}
```

The signed URL is valid for **1 hour**. The audio can be streamed directly from this URL — no auth headers needed. Request a new signed URL when the old one expires.

### Option B: Direct GCS access (for native apps with service account)

If the app has its own GCP service account, it can read directly from the bucket:

- **Bucket:** `noble-imprint-audiobooks`
- **Project:** `noble-imprint-website`
- **Path pattern:** `audio/{slugified-book-path}/{filename}`

The slug function lowercases, strips apostrophes, replaces non-alphanumeric chars with hyphens:
```
"A Library of Classics" → "a-library-of-classics"
"A Pastoral Shelf"      → "a-pastoral-shelf"
"Oration II"            → "oration-ii"
```

Full path example:
```
audio/a-library-of-classics/a-pastoral-shelf/oration-ii/02-chapterone.mp3
```

Required IAM role: `roles/storage.objectViewer` on the `noble-imprint-audiobooks` bucket.

## Manifest Format

The manifest is the entry point — fetch it first, then use it to find audio files for each session.

```json
{
  "bookPath": "series/A Library of Classics/A Pastoral Shelf/Oration II",
  "sessions": [
    {
      "sessionFile": "02-ChapterOne.md",
      "audioFile": "02-chapterone.mp3",
      "ttsFile": "02-chapterone.tts.json",
      "timestampsFile": "02-chapterone.timestamps.json",
      "contentHash": "sha256:322be977294aca88...",
      "chunkHashes": {
        "0": "7eed8cfb3ad9ed9b",
        "1": "cca53b39d9aa1c3a",
        "2": "d2b28a3fe9c0b93d",
        "3": "0419679a28393fe6"
      },
      "chunkCount": 4,
      "durationSeconds": 1158,
      "characterCount": 14824,
      "generatedAt": "2026-05-18T00:45:15.844Z"
    },
    {
      "sessionFile": "03-ChapterTwo.md",
      "audioFile": "03-chaptertwo.mp3",
      "timestampsFile": "03-chaptertwo.timestamps.json",
      "durationSeconds": 573,
      "characterCount": 8269,
      "generatedAt": "2026-05-17T21:41:28.000Z"
    }
  ],
  "totalDurationSeconds": 11358
}
```

### Key fields for the app

| Field | Type | Use |
|-------|------|-----|
| `sessions[].sessionFile` | string | Match to the session filename to find audio for a given chapter |
| `sessions[].audioFile` | string | Filename of the MP3 — use with the signed URL endpoint |
| `sessions[].timestampsFile` | string | Filename of the timestamps JSON — use with the signed URL endpoint |
| `sessions[].durationSeconds` | number | Total duration in seconds — display as chapter length |
| `totalDurationSeconds` | number | Total book audio duration — display as "3h 9m" on the book page |

### Matching sessions to audio

The app displays sessions (chapters) from the Resources repo. To find if a session has audio:

```
manifest.sessions.find(s => s.sessionFile === session.filename)
```

If found, that session has audio. If not found (or if the session is in `skip_sessions`), don't show audio controls.

## Audio Files

- **Format:** MP3, 44.1kHz, 128kbps
- **One file per chapter** (concatenated from ~4,500-char chunks)
- **Duration:** ranges from ~10 minutes (short chapters) to ~58 minutes (long chapters)
- **File size:** roughly 1 MB per minute of audio
- **Streaming:** the signed URL supports HTTP range requests for streaming/seeking

## Timestamps Format

This is the critical file for text synchronization. Each timestamps file contains sentence-level segments aligned to the audio by OpenAI Whisper.

```json
{
  "segments": [
    {
      "start": 0,
      "end": 0.95,
      "text": "Chapter One."
    },
    {
      "start": 0.95,
      "end": 7.72,
      "text": "\"I Have Been Defeated, and Own My Defeat\": Exploring the Dignity of the Ministry Call."
    },
    {
      "start": 7.72,
      "end": 11.97,
      "text": "Section 1: Gregory Addresses His Flight from Ministry."
    },
    {
      "start": 11.97,
      "end": 17.72,
      "text": "Gregory has yielded to the Lord's calling and accepted his ministry post."
    },
    {
      "start": 17.72,
      "end": 24.65,
      "text": "His acceptance of the pastoral office is an act of submission and obedience to the Lord."
    }
  ]
}
```

### Segment fields

| Field | Type | Description |
|-------|------|-------------|
| `start` | number | Start time in seconds (from beginning of the chapter audio) |
| `end` | number | End time in seconds |
| `text` | string | The sentence spoken during this time window |

### Important notes about the text

- **Segments are generated by Whisper transcription**, so the `text` is what Whisper heard, not the original markdown. There may be minor differences in punctuation, capitalization, or word boundaries.
- **Headings have trailing periods** that don't appear in the rendered markdown (e.g., `"Chapter One."` vs rendered `"Chapter One"`). Strip the trailing period when matching against your rendered text.
- **Segments are roughly sentence-level** but Whisper may split or merge sentences differently than you'd expect. Some segments are single words (like `"1."` for a section number).
- **Segments are contiguous** — `end` of one segment equals `start` of the next, with no gaps.
- **Typical segment count:** 80-200 per chapter depending on length.

## Implementing the Player

### Minimum controls

- Play / pause
- Scrubber (progress bar showing current position / total duration)
- Current time and total duration display
- Playback speed (0.75x, 1x, 1.25x, 1.5x)
- Skip backward / forward 15 seconds

### Player lifecycle

1. User navigates to a chapter that has audio (check manifest)
2. Show a play button or audio icon
3. On first play: fetch signed URL, start audio playback
4. Show player controls (sticky bar or embedded in the view)
5. On pause: keep player visible with current position
6. On close/dismiss: stop audio, hide player

### Signed URL expiry

Signed URLs expire after 1 hour. If the user pauses for a long time and resumes, the URL may have expired. Handle this by catching the playback error and requesting a fresh signed URL:

```
1. Try to play/seek using current URL
2. If HTTP 403 or network error:
   a. Fetch new signed URL from /api/audio/url/...
   b. Create new audio source with fresh URL
   c. Seek to the saved position
   d. Resume playback
```

## Implementing Text Sync

This is the most impactful feature — highlighting the sentence being spoken and scrolling the text to follow along.

### Algorithm

```
1. LOAD timestamps JSON when audio starts playing
2. BUILD a mapping from timestamp segments to text elements in your view:
   a. For each segment, take the first ~40 characters of segment.text
   b. Normalize: collapse whitespace, normalize quotes ("" → "), strip trailing period
   c. Search your rendered text elements for one containing that normalized substring
   d. Store: { start, end, textElement }

3. SYNC LOOP (run on every frame or every ~100ms while playing):
   a. Get current playback time
   b. Find the segment where: start <= currentTime < end
   c. If this segment is different from the last highlighted one:
      - Remove previous highlight
      - Highlight the new sentence (see below)
      - Scroll to the highlighted element (smooth scroll, centered)

4. RESPECT MANUAL SCROLL:
   - If the user scrolls manually, pause auto-scrolling for 5 seconds
   - Resume auto-scrolling after 5 seconds of no manual scroll
```

### Highlighting a sentence (not the whole paragraph)

The key challenge: a paragraph element may contain multiple sentences, but you want to highlight just the one being spoken.

**Approach (what the website does):**
1. Walk the text nodes inside the matched element
2. Find the text node containing the sentence text (normalized match)
3. Calculate the character offset within that text node
4. Create a highlight range covering just the sentence characters
5. Apply a highlight style (background color) to that range

In native iOS/Android, you can use attributed strings or spans to highlight a substring within a text view. The logic is the same:
- Find the character range of the sentence within the rendered text
- Apply a background color attribute to that range
- Remove it when the highlight moves to the next sentence

### Matching text to rendered content

The `text` field in timestamps comes from Whisper's transcription of the TTS audio. It closely matches the rendered text but may differ slightly:

| Timestamps text | Rendered text | Difference |
|----------------|---------------|------------|
| `Chapter One.` | `Chapter One` | Trailing period (added for TTS pause) |
| `"I Have Been Defeated"` | `"I Have Been Defeated"` | Smart quotes vs straight quotes |
| `1.` | `1.` | Short segments for section numbers |

**Normalization function** (apply to both timestamp text and rendered text before matching):
```
function normalize(text):
  replace smart quotes (" " ' ') with straight quotes
  replace em/en dashes (— –) with hyphens
  replace ellipsis (…) with "..."
  collapse multiple spaces to single space
  strip trailing period
  trim whitespace
```

Then use substring matching: `renderedText.includes(normalize(segment.text).substring(0, 40))`

## Auto-Advance Between Chapters

When audio for a chapter reaches the end:

1. Check if there is a next chapter with audio (next entry in the manifest, or next session in the book's session list)
2. If yes: navigate to that chapter and auto-start playback
3. If no: stop playback (end of book)

Store an "auto-play" flag (e.g., in UserDefaults / SharedPreferences) when auto-advancing so the next chapter view knows to start playing immediately.

---

## Resume Playback

Save the playback position periodically (every few seconds) so the user can resume where they left off:

**Storage key:** `audio-pos:{bookPath}/{audioFile}`
**Value:** current time in seconds (e.g., `542.3`)

On page/view load:
1. Check for a saved position for this chapter
2. If found and within reasonable bounds (> 0 and < duration - 5), seek to that position
3. Don't auto-play on resume (unless arriving from auto-advance)

Clear the saved position when the chapter finishes playing.

## Offline Support

For offline playback, the app can download and cache audio files:

1. **Download the manifest** and cache it locally
2. **Download MP3 files** for selected chapters (or the whole book)
3. **Download timestamps files** alongside their MP3s
4. Serve all files from local storage during playback

**Storage estimate per book:**
- Oration II (6 chapters, 3.2 hours): ~180 MB of MP3 files + ~1 MB of JSON
- A typical chapter: 10–60 MB depending on length

When online, check the manifest's `generatedAt` timestamps against your cached versions to detect updated audio.

## Reference: Website Implementation

The website's implementation can serve as a reference. The key files are:

| File | Location | What it does |
|------|----------|--------------|
| `audio-player.js` | `Noble-Imprint-Resource-Website/src/public/js/audio-player.js` | Full player + text sync in ~250 lines of vanilla JS |
| `audio.js` | `Noble-Imprint-Resource-Website/src/server/audio.js` | Server-side GCS manifest loading + signed URL generation |
| `style.css` | `Noble-Imprint-Resource-Website/src/public/css/style.css` | Player bar + highlight styles (search for `.audio-`) |
| `session.ejs` | `Noble-Imprint-Resource-Website/src/views/session.ejs` | Player HTML template (search for `audio-fab`) |

The website player:
- Uses HTML5 `<audio>` element for playback
- Fetches signed URLs lazily on first play
- Runs a `requestAnimationFrame` loop for sync
- Wraps sentences in `<mark class="audio-highlight">` using the DOM Range API
- Saves position to `localStorage`
- Auto-advances using a `localStorage` flag

## Reference: API Endpoints on the Website

All endpoints are public (no auth required).

| Method | Endpoint | Returns |
|--------|----------|---------|
| `GET` | `/api/audio/manifest/{bookPath}` | Manifest JSON with all sessions, durations, file names |
| `GET` | `/api/audio/url/{bookPath}/{filename}` | `{ "url": "signed-gcs-url" }` valid for 1 hour |
| `POST` | `/api/refresh-audio` | Clears the server's audio manifest cache |

The `{bookPath}` is the path within the Resources repo **without** the `series/` prefix:
```
A Library of Classics/A Pastoral Shelf/Oration II
```

URL-encode spaces: `A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II`
