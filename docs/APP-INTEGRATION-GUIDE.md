## What Already Exists

Audio is generated automatically by a pipeline that runs when book content changes. The pipeline produces three files per chapter, stored in Google Cloud Storage:

| File | Example | Purpose |
|------|---------|---------|
| **Chapter MP3** | `02-chapterone.mp3` | The narrated audio for that chapter |
| **Timestamps JSON** | `02-chapterone.timestamps.json` | Sentence-level start/end times mapped to text |
| **TTS JSON** | `02-chapterone.tts.json` | The preprocessed text that was sent to the TTS engine (for debugging) |

A **manifest** (`manifest.json`) indexes all chapters with their file names, durations, character counts, and content hashes.

All files are in a GCS bucket (`noble-imprint-audiobooks`) and are **not publicly accessible**. They must be accessed via signed URLs (1-hour expiry) generated by the website's API, or directly via GCS SDK with appropriate service account credentials.

## Dynamic Audio Detection Strategy

**The core principle: the app should never hardcode which books have audio.** Audio support for any book can be added server-side at any time, and the app should pick it up automatically without a rebuild or app store update.

### How it works

Audio availability is determined entirely at runtime by checking for a manifest file in GCS. The app does not need to know in advance which books have audio. The flow:

```
App launches or user opens a book
  │
  ├── App already has the book's content (sessions, text, etc.)
  │
  ├── App asks: "Does this book have audio?"
  │   └── GET /api/audio/manifest/{bookPath}
  │       ├── 200 + JSON → YES, audio exists. Cache the manifest.
  │       └── 404 → NO audio for this book. Don't show audio controls.
  │
  ├── If YES: show audio icon/controls on sessions listed in manifest
  │   └── For each session, check: manifest.sessions.find(s => s.sessionFile === thisSession)
  │       ├── Found → audio available for this chapter. Show play button.
  │       └── Not found → this chapter has no audio (skipped). No play button.
  │
  └── When user taps play:
      ├── GET /api/audio/url/{bookPath}/{audioFile} → signed URL (1 hour)
      ├── GET /api/audio/url/{bookPath}/{timestampsFile} → signed URL for timestamps
      ├── Fetch timestamps JSON from the signed URL
      ├── Start audio playback from the signed MP3 URL
      └── Begin text sync using timestamps
```

### What this means for the app developer

1. **Build the audio player UI once** — it works for any book, present or future.

2. **On every book load, check the manifest endpoint.** This is a lightweight JSON fetch (~2 KB). Cache it for the session or for a few minutes. If it returns 404, the book has no audio — hide all audio UI.

3. **Don't gate audio features behind app versions.** If a new book gets audio added six months from now, every installed app that already has that book will automatically show the audio player the next time the user opens it — no app update needed.

4. **The manifest is the source of truth.** It tells you:
   - Which sessions have audio (`sessions[].sessionFile`)
   - How long each chapter is (`sessions[].durationSeconds`)
   - Total book duration (`totalDurationSeconds`)
   - When audio was last generated (`sessions[].generatedAt`)
   - File names for audio and timestamps

5. **Session filenames are the join key.** The app already knows its session files (e.g., `02-ChapterOne.md`). The manifest uses the same filenames. Match on `sessionFile` to connect audio to content.

### Recommended caching strategy

| What | Cache duration | Storage |
|------|---------------|---------|
| Manifest JSON | 5–10 minutes (or until app backgrounded) | Memory |
| Signed URLs | Up to 1 hour (expiry is in the URL) | Memory |
| Timestamps JSON | Until manifest changes (keyed by `generatedAt`) | Disk |
| MP3 files (streaming) | Don't cache unless offline mode | — |
| MP3 files (offline) | Until manifest `generatedAt` changes | Disk |

### Detecting updates after content changes

When book content changes and audio is regenerated:
- The manifest's `generatedAt` timestamps update
- The `contentHash` for affected sessions changes

The app can detect this by periodically re-fetching the manifest (or on each book open) and comparing `generatedAt`. If a session's timestamp changed, its audio and timestamps files are new — invalidate any cached copies.

### Example: adding audio to an existing book

Suppose a new book "Venture" has no audio yet, while "Oration II" and "HomeStead" already have audiobooks. When audio is added for Venture:

1. We add `audiobook.enabled: true` to Venture's `meta.json`
2. The pipeline generates MP3s and timestamps, uploads to GCS
3. A manifest appears at `audio/passage-series/venture/manifest.json`
4. Next time a user opens Venture in the app, the manifest fetch returns 200 instead of 404
5. The app shows the audio player — no app update required

## How to Determine if a Book Has Audio

Each book has a `meta.json` in the Resources repo. If the book has audiobook generation enabled, it contains:

```json
{
  "title": "Oration II",
  "subtitle": "In Defense of His Flight to Pontus",
  "audiobook": {
    "enabled": true
  }
}
```

**Check:** `meta.audiobook && meta.audiobook.enabled === true`

If `skip_sessions` is present, sessions listed there do NOT have audio — don't show a play button for those. Most books include all sessions.

## Fetching Audio Files

### Option A: Use the website API (simplest)

The website exposes two endpoints that handle GCS authentication and return signed URLs:

**Get the manifest:**
```
GET https://resources.noblecollective.org/api/audio/manifest/{bookPath}
```

Example:
```
GET https://resources.noblecollective.org/api/audio/manifest/A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II
```

Returns the full manifest JSON (see format below).

**Get a signed URL for any file:**
```
GET https://resources.noblecollective.org/api/audio/url/{bookPath}/{filename}
```

Example:
```
GET https://resources.noblecollective.org/api/audio/url/A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II/02-chapterone.mp3
```

Returns:
```json
{
  "url": "https://storage.googleapis.com/noble-imprint-audiobooks/audio/a-library-of-classics/a-pastoral-shelf/oration-ii/02-chapterone.mp3?GoogleAccessId=...&Expires=...&Signature=..."
}
```

The signed URL is valid for **1 hour**. The audio can be streamed directly from this URL — no auth headers needed. Request a new signed URL when the old one expires.

### Option B: Direct GCS access (for native apps with service account)

If the app has its own GCP service account, it can read directly from the bucket:

- **Bucket:** `noble-imprint-audiobooks`
- **Project:** `noble-imprint-website`
- **Path pattern:** `audio/{slugified-book-path}/{filename}`

The slug function lowercases, strips apostrophes, replaces non-alphanumeric chars with hyphens:
```
"A Library of Classics" → "a-library-of-classics"
"A Pastoral Shelf"      → "a-pastoral-shelf"
"Oration II"            → "oration-ii"
"L'Appel du Christ"     → "lappel-du-christ"
```

Full path example:
```
audio/a-library-of-classics/a-pastoral-shelf/oration-ii/02-chapterone.mp3
```

Required IAM role: `roles/storage.objectViewer` on the `noble-imprint-audiobooks` bucket.

## Manifest Format

The manifest is the entry point — fetch it first, then use it to find audio files for each session.

```json
{
  "bookPath": "series/A Library of Classics/A Pastoral Shelf/Oration II",
  "sessions": [
    {
      "sessionFile": "02-ChapterOne.md",
      "audioFile": "02-chapterone.mp3",
      "ttsFile": "02-chapterone.tts.json",
      "timestampsFile": "02-chapterone.timestamps.json",
      "contentHash": "sha256:322be977294aca88...",
      "chunkHashes": {
        "0": "7eed8cfb3ad9ed9b",
        "1": "cca53b39d9aa1c3a",
        "2": "d2b28a3fe9c0b93d",
        "3": "0419679a28393fe6"
      },
      "chunkCount": 4,
      "durationSeconds": 1158,
      "characterCount": 14824,
      "generatedAt": "2026-05-18T00:45:15.844Z"
    },
    {
      "sessionFile": "03-ChapterTwo.md",
      "audioFile": "03-chaptertwo.mp3",
      "timestampsFile": "03-chaptertwo.timestamps.json",
      "durationSeconds": 573,
      "characterCount": 8269,
      "generatedAt": "2026-05-17T21:41:28.000Z"
    }
  ],
  "totalDurationSeconds": 11358
}
```

### Key fields for the app

| Field | Type | Use |
|-------|------|-----|
| `sessions[].sessionFile` | string | Match to the session filename to find audio for a given chapter |
| `sessions[].audioFile` | string | Filename of the MP3 — use with the signed URL endpoint |
| `sessions[].timestampsFile` | string | Filename of the timestamps JSON — use with the signed URL endpoint |
| `sessions[].durationSeconds` | number | Total duration in seconds — display as chapter length |
| `totalDurationSeconds` | number | Total book audio duration — display as "3h 9m" on the book page |

### Matching sessions to audio

The app displays sessions (chapters) from the Resources repo. To find if a session has audio:

```
manifest.sessions.find(s => s.sessionFile === session.filename)
```

If found, that session has audio. If not found (or if the session is in `skip_sessions`), don't show audio controls.

## Audio Files

- **Format:** MP3, 44.1kHz, 128kbps
- **One file per chapter** (concatenated from ~800-char chunks, no silence gaps between chunks)
- **Duration:** ranges from ~10 minutes (short chapters) to ~390 minutes (very long chapters like HomeStead Part One)
- **File size:** roughly 1 MB per minute of audio
- **Streaming:** the signed URL supports HTTP range requests for streaming/seeking

## Timestamps Format

This is the critical file for text synchronization. Each timestamps file contains sentence-level segments with character-level timing from ElevenLabs and structural indices for reliable element lookup.

```json
{
  "segments": [
    {
      "start": 0,
      "end": 0.95,
      "blockIndex": 0,
      "sentenceIndex": 0,
      "text": "Chapter six"
    },
    {
      "start": 0.95,
      "end": 7.72,
      "blockIndex": 1,
      "sentenceIndex": 0,
      "text": "\"Set in the midst betwixt two fears\": accepting God's call with humility and duty"
    },
    {
      "start": 7.72,
      "end": 11.97,
      "blockIndex": 2,
      "sentenceIndex": 0,
      "text": "Section 102: Mutual affection brought Gregory back"
    },
    {
      "start": 11.97,
      "end": 17.72,
      "blockIndex": 3,
      "sentenceIndex": 0,
      "text": "Gregory summarizes that the previous account has been a defense of why he fled his ministry call."
    },
    {
      "start": 17.72,
      "end": 24.65,
      "blockIndex": 3,
      "sentenceIndex": 1,
      "text": "He knows it caused himself and the people pain, but he felt the need—\"with irresistible force\"—to not accept this post of spiritual leadership."
    }
  ]
}
```

### Segment fields

| Field | Type | Description |
|-------|------|-------------|
| `start` | number | Start time in seconds (from beginning of the chapter audio). Character-level precision from ElevenLabs. |
| `end` | number | End time in seconds |
| `blockIndex` | number | Sequential index of the block (heading or paragraph) as counted by the preprocessor. **Important:** do not use as a direct array index into rendered DOM elements — the rendered page may have extra elements (from Question tags, tables, list items, etc.). Use `text` matching instead (see below). |
| `sentenceIndex` | number | Index of the sentence within that block, 0-based. A heading is always sentence 0. A paragraph with 3 sentences has indices 0, 1, 2. |
| `text` | string | The exact source sentence text from the preprocessor. Use this to find the correct element in your rendered content via text matching. |

### How blockIndex and sentenceIndex work

The preprocessor assigns a sequential `blockIndex` to every heading and paragraph in the chapter. Within each block, sentences are split on `.!?` followed by whitespace (excluding ellipsis `...`) and assigned a `sentenceIndex`.

Example for a chapter with: H1, H3 subtitle, H2 section heading, paragraph (3 sentences), H2 section heading:

| blockIndex | sentenceIndex | Element | Text |
|-----------|---------------|---------|------|
| 0 | 0 | `<h1>` | "Chapter six" |
| 1 | 0 | `<h3>` | "Set in the midst..." |
| 2 | 0 | `<h2>` | "Section 102: Mutual affection..." |
| 3 | 0 | `<p>` | "Gregory summarizes that..." |
| 3 | 1 | `<p>` | "He knows it caused..." |
| 3 | 2 | `<p>` | "The mutual love between..." |
| 4 | 0 | `<h2>` | "Section 103: Conquered..." |

Note: heading text in segments is sentence-cased (e.g., "Mutual affection" not "Mutual Affection") and has no trailing period.

Multiple segments share the same `blockIndex` when they're different sentences within the same paragraph.

### Important notes

- **Text comes from our preprocessor**. For paragraphs, it matches the rendered markdown content directly. For headings, the text is sentence-cased for TTS — use case-insensitive matching when joining to rendered content.
- **Headings are sentence-cased** for TTS (e.g., `"Responsibility commitment"` not `"Responsibility Commitment"`), with proper nouns whitelisted. The original display text is stored in `block.displayText` for web reader matching. Headings get SSML `<break>` pauses (H1: 2s, H2: 1.5s, H3-H6: 1s) instead of trailing periods. Match heading segments against rendered text using the display text, not the TTS text.
- **Timing comes from ElevenLabs** character-level alignment — generated alongside the audio for exact accuracy. No silence gaps between chunks — timestamps are contiguous with zero cumulative drift.
- **Segments are contiguous** — timestamps cover the full chapter duration with no gaps.
- **blockIndex is a hint, not an index.** The preprocessor and renderer may count elements differently. Always match by `text`, using `blockIndex` only as a starting position for the search.
- **Typical count:** 60–4,400+ segments per chapter depending on length. HomeStead Part One has 4,425 segments across 390 minutes.
- **Numbered oration paragraphs** (e.g., "103.") have a paragraph break inserted after the number to produce a TTS pause: `"103.\n\nIn the next place..."`. This does not affect the rendered content.

## Implementing the Player

### Minimum controls

- Play / pause
- Scrubber (progress bar showing current position / total duration)
- Current time and total duration display
- Playback speed (0.75x, 1x, 1.25x, 1.5x)
- Skip backward / forward 15 seconds

### Player lifecycle

1. User navigates to a chapter that has audio (check manifest)
2. Show a play button or audio icon
3. On first play: fetch signed URL, start audio playback
4. Show player controls (sticky bar or embedded in the view)
5. On pause: keep player visible with current position
6. On close/dismiss: stop audio, hide player

### Signed URL expiry

Signed URLs expire after 1 hour. If the user pauses for a long time and resumes, the URL may have expired. Handle this by catching the playback error and requesting a fresh signed URL:

```
1. Try to play/seek using current URL
2. If HTTP 403 or network error:
   a. Fetch new signed URL from /api/audio/url/...
   b. Create new audio source with fresh URL
   c. Seek to the saved position
   d. Resume playback
```

## Implementing Text Sync

This is the most impactful feature — highlighting the sentence being spoken and scrolling the text to follow along.

### Algorithm

```
1. LOAD timestamps JSON eagerly on page load (not just on first play).
   This allows heading audio icons and scrubber markers to appear immediately.

2. BUILD a segment map by matching each timestamp segment to a rendered element.

   IMPORTANT: Do NOT use blockIndex as a direct array index. The preprocessor
   and renderer may produce different element counts (the preprocessor strips
   Question tags, tables, list items, etc. but the renderer keeps them). Instead,
   match segments to elements by TEXT CONTENT:

   a. Collect all block elements (h1-h6, p) in document order
   b. Pre-normalize the text content of each element (lowercase, strip quotes/punctuation)
   c. For each segment, use blockIndex as a HINT for approximate position,
      then search nearby elements for one whose text contains the segment's text.
      For headings, match against the display text (original casing), not the
      sentence-cased TTS text.
   d. Track which elements have been matched to avoid duplicate assignments
      (handles repeated headings like "Overview" that appear in every section)

3. FOR EACH timestamp segment, resolve its sentence within the matched element:
   - sentenceIndex → the Nth sentence within that block's text
     (split on .!? followed by whitespace — same rule the preprocessor uses)

4. SYNC LOOP (run on every frame or every ~100ms while playing):
   a. Get current playback time
   b. Find the segment where: start <= currentTime < end
   c. If no segment matches (gap between segments): clear the highlight
   d. If this segment is different from the last highlighted one:
      - Remove previous highlight
      - Highlight the sentence at the matched element + sentenceIndex
      - If auto-scroll is engaged: scroll to that element

5. SCROLL BEHAVIOR:
   - Auto-scroll follows the highlighted sentence by default
   - When the user manually scrolls away, auto-scroll disengages and a
     "Jump to audio location" link appears above the player
   - Tapping the link scrolls back to the highlight and re-engages auto-scroll
   - Skip (±15s) and scrub always force-scroll to the new position

6. HEADING AUDIO ICONS:
   - After timestamps load, add clickable headphone icons to h1-h6 headings
   - Clicking an icon starts playback (if not already playing) and seeks to
     that heading's exact timestamp
```

### Finding the element (text matching, not blockIndex counting)

**Use text matching, not direct blockIndex counting.** The rendered page may have more elements than the preprocessor counted (due to Question tags, table cells, list items rendered as paragraphs, etc.). A direct `blocks[blockIndex]` lookup can be off by dozens of elements in long chapters.

The recommended approach:

```
// Pre-compute normalized text for all block elements (once)
blocks = getAllBlockElements()  // h1-h6, p in document order
blockTexts = blocks.map(el => normalize(el.textContent))

// For each segment, search near the hinted position
offsetAdjust = 0
matchedElements = Set()

for each segment:
    hint = segment.blockIndex + offsetAdjust
    needle = normalize(segment.text).first30Chars()

    // Search forward from hint, then backward if needed
    for i from hint to end:
        if blockTexts[i].contains(needle) and not already matched (for sentenceIndex 0):
            element = blocks[i]
            offsetAdjust = i - segment.blockIndex  // calibrate for future segments
            break

    // Mark element as matched (prevents duplicate assignment for repeated headings)
    if sentenceIndex == 0: matchedElements.add(i)
```

This handles:
- **Extra DOM elements** — the offset is tracked and calibrated per match
- **Repeated headings** — "Overview", "Conclusion", etc. appearing multiple times won't double-match
- **Long chapters** — HomeStead Part One has 1544 DOM elements vs 1460 preprocessor blocks (84 extra)

### Finding the sentence within the element (sentenceIndex)

Once you have the correct element, split its text content into sentences using the same rule the preprocessor uses:

```
sentences = element.textContent.split(/(?<=[.!?])(?<!\.\.\.)(?<!\.\.\.\s)\s+/)
targetSentence = sentences[segment.sentenceIndex]
```

Then find that sentence's character position within the element's full text:

```
startOffset = element.textContent.indexOf(targetSentence)
endOffset = startOffset + targetSentence.length
```

### Highlighting a sentence (not the whole paragraph)

**On the web (what the website does):**
1. Walk the text nodes inside the element to find start/end character offsets
2. Create a `Range` spanning those offsets (handles text nodes split by `<strong>`, `<em>`, etc.)
3. Call `range.getClientRects()` to get pixel rectangles for the sentence
4. Position transparent overlay divs at those rectangles (no DOM modification)
5. Clear overlays when the highlight moves to the next sentence

**In native iOS:**
- Use `NSAttributedString` with a background color attribute on the character range
- Or use `UITextView` with `NSRange` highlighting

**In native Android:**
- Use `SpannableString` with `BackgroundColorSpan` on the character range
- Or use `TextView` with `setSelection()` for the range

### Fallback behavior

If `sentenceIndex` is 0 and the block has only one sentence (most headings), highlight the entire element — no sentence splitting needed.

If the sentence split produces an unexpected count (edge case with abbreviations like "St. Paul"), fall back to highlighting the whole element. This is rare and still provides a good user experience.

## Auto-Advance Between Chapters

When audio for a chapter reaches the end:

1. Check if there is a next chapter with audio (next entry in the manifest, or next session in the book's session list)
2. If yes: navigate to that chapter and auto-start playback
3. If no: stop playback (end of book)

Store an "auto-play" flag (e.g., in UserDefaults / SharedPreferences) when auto-advancing so the next chapter view knows to start playing immediately.

---

## Resume Playback

Save the playback position periodically (every few seconds) so the user can resume where they left off:

**Storage key:** `audio-pos:{bookPath}/{audioFile}`
**Value:** current time in seconds (e.g., `542.3`)

On page/view load:
1. Check for a saved position for this chapter
2. If found and within reasonable bounds (> 0 and < duration - 5), seek to that position
3. Don't auto-play on resume (unless arriving from auto-advance)

Clear the saved position when the chapter finishes playing.

## Offline Support

For offline playback, the app can download and cache audio files:

1. **Download the manifest** and cache it locally
2. **Download MP3 files** for selected chapters (or the whole book)
3. **Download timestamps files** alongside their MP3s
4. Serve all files from local storage during playback

**Storage estimate per book:**
- Oration II (6 chapters, 3.2 hours): ~180 MB of MP3 files + ~1 MB of JSON
- A typical chapter: 10–60 MB depending on length

When online, check the manifest's `generatedAt` timestamps against your cached versions to detect updated audio.

## What You Reuse vs What You Build

### Reuse from the server (no rebuilding)

Everything below is generated by the pipeline and stored in GCS. The app consumes these files as-is:

| Asset | What it is | How the app uses it |
|-------|-----------|-------------------|
| **MP3 files** | Narrated chapter audio (44.1kHz, 128kbps) | Stream via signed URL or download for offline |
| **Timestamps JSON** | `blockIndex`, `sentenceIndex`, `start`, `end` per sentence | Drive the text sync highlighting |
| **Manifest JSON** | Index of all sessions with durations and filenames | Discover which chapters have audio, show durations |
| **Signed URL API** | Website endpoints that generate 1-hour GCS access URLs | Fetch audio and timestamp files without GCS credentials |

### Build in native code

These are the components the app developer needs to implement:

| Component | What to build | Guidance |
|-----------|--------------|---------|
| **Audio player** | Native playback (AVPlayer / MediaPlayer) with play/pause, scrubber, speed, skip ±15s | See [Implementing the Player](#implementing-the-player) |
| **Block element mapping** | Match segments to rendered views by **text content** (not blockIndex counting). Use blockIndex as a hint, search nearby for matching text. Track matched views to handle repeated headings. | See [Finding the element](#finding-the-element-text-matching-not-blockindex-counting) |
| **Sentence splitting** | Split a paragraph's text on `.!?` followed by whitespace, excluding ellipsis (`...`) — same regex as the preprocessor: `split(/(?<=[.!?])(?<!\.\.\.)(?<!\.\.\.\s)\s+/)` | See [Finding the sentence](#finding-the-sentence-within-the-element-sentenceindex) |
| **Sentence highlighting** | Apply background color to the character range of the target sentence within the correct text view | iOS: `NSAttributedString` with `NSBackgroundColorAttributeName`. Android: `SpannableString` with `BackgroundColorSpan`. |
| **Sync loop** | Timer or display link (~100ms) that reads `currentTime`, finds active segment, updates highlight | See [Algorithm](#algorithm) |
| **Scroll behavior** | Smooth scroll to the highlighted element. When user scrolls away, show "Jump to audio" link instead of force-scrolling back. | See step 5 in the [Algorithm](#algorithm) |
| **Heading audio icons** | Clickable headphone icons on h1-h6 headings that seek audio to that heading's timestamp | See step 6 in the [Algorithm](#algorithm) |
| **H2 scrubber markers** | Tick marks on the progress bar at H2 heading positions, clickable to seek | Use H2 segments from timestamps |
| **Resume playback** | Save/restore position in UserDefaults / SharedPreferences | See [Resume Playback](#resume-playback) |
| **Auto-advance** | Navigate to next chapter and auto-play when current chapter ends | See [Auto-Advance](#auto-advance-between-chapters) |
| **Signed URL refresh** | Catch 403 errors (expired URL), request a fresh one, resume playback | See [Signed URL expiry](#signed-url-expiry) |
| **Offline download** (optional) | Download MP3 + timestamps for selected chapters, serve locally | See [Offline Support](#offline-support) |

### What you do NOT need to build

- Audio generation — handled by the pipeline
- Timestamp alignment — handled by ElevenLabs character-level timestamps (generated alongside audio)
- Text preprocessing — handled by the pipeline
- GCS storage management — handled by the pipeline
- Content change detection — handled by the pipeline

- Matches segments to DOM elements by **text content** (not blockIndex counting) — handles extra DOM elements from Question tags, tables, etc.
- Uses `sentenceIndex` to split the element's text and find the target sentence
- Creates a `Range` over the sentence and uses `getClientRects()` to position transparent highlight overlays (no DOM modification — works across `<strong>`, `<em>` boundaries)
- Clears highlight when current time falls in a gap between segments
- Autoscroll follows the highlighted sentence, accounting for sticky header and player bar height
- "Jump to audio location" pill link appears when user scrolls away — tapping re-engages autoscroll
- Skip ±15s and scrub force-update the highlight and scroll immediately
- H2 section markers on the scrubber bar — clickable tick marks at exact heading positions
- Clickable headphone icons on h1-h6 headings — seeks audio to that heading's timestamp
- Mobile: expandable player bar (tap chevron for speed control and ±15s skip)
- Saves position to `localStorage`
- Auto-advances to the next chapter using a `localStorage` flag
- Floating headphones icon expands to sticky bottom bar when playing

## Reference: API Endpoints on the Website

All endpoints are public (no auth required).

| Method | Endpoint | Returns |
|--------|----------|---------|
| `GET` | `/api/audio/manifest/{bookPath}` | Manifest JSON with all sessions, durations, file names |
| `GET` | `/api/audio/url/{bookPath}/{filename}` | `{ "url": "signed-gcs-url" }` valid for 1 hour |
| `POST` | `/api/refresh-audio` | Clears the server's audio manifest cache |

The `{bookPath}` is the path within the Resources repo **without** the `series/` prefix:
```
A Library of Classics/A Pastoral Shelf/Oration II
```

URL-encode spaces: `A%20Library%20of%20Classics/A%20Pastoral%20Shelf/Oration%20II`
